# fix-b6e4be97026987c3 — Two conflicting `/start` handlers — E3T1.ts shadows start.ts at runtime

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 SPLT

`src/handlers/E3T1.ts` and `src/handlers/start.ts` both register `command("start", ...)` with different replies (E3T1 sends a plain welcome, start.ts sends a welcome with an inline menu keyboard). `buildBot()` sorts handler files alphabetically, so `E3T1.ts` is loaded before `start.ts`. grammY's linear middleware chain means the first registered handler that matches consumes the update — E3T1's `/start` handler wins every time, and the `start.ts` menu is **never reachable** in production. The specs in `tests/specs/start.json` and `tests/specs/T02.json` expect the menu reply and pass only because the harness runs each spec against a fresh isolated bot — they misrepresent actual runtime behavior. Fix: remove the duplicate `/start` from E3T1.ts or start.ts (or merge them into one handler).
